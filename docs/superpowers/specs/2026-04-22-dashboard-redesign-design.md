# Dashboard Redesign — Design Spec

**Date:** 2026-04-22
**Status:** Approved, ready for implementation planning
**Scope:** Redesign the investment agent's HTML dashboard as a modern Vue 3 application in a separate repository, consuming portfolio data through a versioned JSON contract.

---

## 1. Overview

The current dashboard is a single self-contained HTML file, around 360 lines of static output generated by a 494-line Python module that interleaves data fetching with markup strings. It works, but it is visually plain, hard to iterate on, and mixes presentation with data extraction inside the main agent repository.

This design replaces that dashboard with a proper Vue 3 application living in its own GitHub repository. The main agent repository keeps owning the data; the dashboard repository owns only the presentation. A single JSON file acts as the contract between the two. This gives the user a cleaner modern interface on both desktop and phone, while preparing the dashboard to be deployed live to Cloudflare Pages in a later phase without any rewrites.

The visual direction is a light, clean, card-based aesthetic inspired by modern consumer fintech products. Content flows in a single centered column on desktop and stacks naturally on phones. The same codebase handles both viewports through responsive utility classes.

---

## 2. Goals

The redesigned dashboard must present a clearer, more insight-rich view of the portfolio than the current one. Specifically, it must surface the total portfolio value with its profit-and-loss against the VOO benchmark, a visual breakdown of how capital is allocated across positions and cash, the daily briefing from Claude along with the bilingual market education content, a preview of the plan that Claude produced during Run 1 today, a complete filterable history of every trade ever made with per-ticker realized profit grouping and CSV export, and a set of performance statistics covering win rate, average winner and loser, maximum drawdown, daily volatility, best trade, and total realized profit.

The redesign must preserve the existing invariant that trade execution is sacred. Nothing about dashboard generation or export may interfere with Run 2's ability to read its plan, execute trades, and record them to the database. The dashboard is downstream of trading; it never gates it.

The redesign must also leave the door open to deploying the dashboard publicly on Cloudflare Pages behind Cloudflare Access at a later point. That later phase is explicitly out of scope for this spec, but nothing in this spec may contradict it.

---

## 3. Non-Goals

Live deployment to Cloudflare Pages, Cloudflare Access authentication configuration, automated build pipelines, and remote hosting of the portfolio data are all explicitly deferred to a future phase and are not covered here.

Interactivity beyond client-side filtering and CSV export is not in scope. The dashboard does not let the user place trades, modify positions, or change agent behavior. It is a read-only view of what the agent has done and what it plans to do.

Migration of the portfolio SQLite database to any cloud database is also deferred. The main agent continues to use the local SQLite file exactly as it does today.

Live per-position profit-and-loss in the hero area is not displayed prominently. The data is computed and available in the positions detail rows at market value, but the top of the page emphasizes portfolio-level outcomes rather than position-by-position figures.

Sector or theme exposure breakdowns and a dividend event log are not surfaced in the UI for this version, although dividend event data is carried through in the JSON payload so a future version can display it without schema changes.

A progress-to-target meter against the thirty-day fifteen-percent goal is not displayed. The raw data is sufficient to compute it later if wanted.

---

## 4. Architecture

Two repositories are involved. The existing investmentAgent repository continues to own the portfolio database, the agent logic, and the run scheduling. A new repository called investmentAgentDashboard is created on GitHub and cloned to the same parent directory on disk, next to the main repository. It owns the presentation layer entirely.

Between them sits a single file, a JSON document that describes the current state of the portfolio and everything needed to render the dashboard. The main repository writes this file. The dashboard repository reads it. Neither side knows anything else about the other.

The main repository gains one new Python module whose only job is to produce this JSON file. It reads from the existing SQLite tables and the existing Run 1 plan file, computes derived statistics, and writes the JSON to a fixed location inside the dashboard repository's public folder. Immediately after writing, it runs a small git helper that stages the file, commits it with a timestamped message, and pushes to the remote. If the git push fails, the run continues without error; the next run will retry. If the dashboard repository folder does not exist on disk, the exporter logs a warning and skips, still without disturbing Run 2.

This new export step runs at the end of Run 2, after trades have already been executed and the trade log has been written. Placing it last guarantees that any failure in export cannot affect trading.

The dashboard repository is a standard Vue 3 application scaffolded with Vite and configured with TypeScript and Tailwind. When served, it loads the JSON file from its own public folder as a static asset and reactively renders the page around its contents. There is no server component, no API, and no build-time baking of portfolio values into the bundle. The JSON file is fetched at runtime and can be updated by simply committing a new copy.

This layout means the dashboard repository is already structurally ready for Cloudflare Pages deployment. When the later phase arrives, the only change needed is to point the fetch at a remote URL instead of the local public folder, and to add a step that uploads the JSON to that remote location. No schema change, no code rewrite.

## 5. Data Contract

The JSON file contains a fixed set of top-level sections, each serving a specific part of the dashboard. The field names are stable once published. New fields may be added over time; existing fields are never renamed or removed without bumping a schema version integer that also lives at the top of the file. The dashboard checks this version on load and shows a warning banner if the number is higher than it was built against, but it still renders on a best-effort basis.

The first section is metadata. It carries the schema version, a timestamp indicating when the file was generated, the current date in eastern time, and an identifier for the run that produced it. This lets the dashboard footer display a generated-at time and lets future tooling diff two snapshots.

The second section is an account summary. It contains the total portfolio value, the cash balance, the original starting cash amount, the overall profit in dollars and percent, and the gap against the VOO benchmark in dollars and percent.

The third section lists current positions. Each position carries its ticker, number of shares, average purchase cost, current market price fetched once at export time, resulting market value, dollar and percent profit, and the percentage of the total portfolio that position represents. When a price fetch fails for any reason, the current price is left empty and any derived fields that depend on it are also empty, but the position still appears.

The fourth section is a pre-computed allocation breakdown for the donut chart. It lists each position's share of total market value as a percentage, plus cash's percentage. The numbers are rounded to sum to exactly one hundred so the chart renders cleanly.

The fifth section is the history of daily snapshots. Each entry carries the date, the total portfolio value that day, the cash balance, and the profit percent. Entries go back as far as the database has them.

The sixth section is the parallel VOO benchmark series. It carries the current VOO share count, the current VOO price, the current benchmark total value, the total amount deposited to the benchmark account, and a per-day snapshot series that mirrors the portfolio one so the agent-versus-VOO chart can plot them together.

The seventh section is the complete trade history. Every trade ever recorded appears here. Each carries its id, timestamp, action (buy or sell), ticker, share count, price, total dollar amount, and the reasoning text that was saved alongside. The list is not truncated. The dashboard filters and groups it client-side.

The eighth section is today's plan, lifted from the Run 1 plan JSON file. It contains the decisions list with each decision's action, ticker, share count, and reasoning, plus the flag indicating whether new buys were skipped today and Claude's briefing text. When no plan file exists, this section is empty and the dashboard shows a "no plan today" placeholder.

The ninth section carries the daily educational content produced by Claude, the market education summary with its English and Chinese versions and source citations, and the daily lesson term with its bilingual explanation. These come from the existing Feature 001 fields in the Run 1 output and flow straight through without transformation.

The tenth section holds derived statistics computed in Python so the dashboard never has to do math. It carries the win rate, the count of winning and losing round trips, the average dollar amount of winning trades, the average dollar amount of losing trades, the single best and single worst realized trades with their tickers and amounts, the maximum drawdown percent measured peak-to-trough across snapshot history, the daily return volatility expressed as a standard deviation, the total realized profit across all closed positions, and the realized profit broken out per ticker.

An eleventh section carries dividend events from the existing database table, each with its date, ticker, share count held at the time, dollar amount per share, and total dollar credit. This section is populated so that a later version can render a dividend log without any schema change; the current dashboard does not display it.

The file is written atomically. The exporter writes to a temporary file first, then renames it into place. This prevents the dashboard from ever reading a half-written file if the export process is interrupted.

## 6. UI Layout

On desktop, the dashboard is a single centered column approximately seven hundred twenty pixels wide, with generous white space around it. On a phone, the same column fills the viewport with sixteen pixels of side padding. No other layouts exist. There is no sidebar, no tabs, no drawer. The user scrolls from top to bottom through the sections in order.

The header sits at the top. It shows the application name on the left and the current date along with the last-updated time on the right. Below it sits the hero card, whose job is to answer the question "how am I doing" in a single glance. The hero leads with the total portfolio value in a very large weight, followed by a colored pill showing the profit percent and dollar amount. Below the pill, three small equally-weighted metric cards show the cash balance, the percent gap versus VOO, and the count of open positions.

Directly below the hero is the equity curve chart. It plots the agent's total value over time as a solid line and the VOO benchmark over the same period as a dashed line. A small legend underneath identifies which is which.

Next is the allocation and positions card. On desktop, this splits into two side-by-side regions: the donut chart on the left and a list of positions on the right. On phone, the donut sits above the list. Each position row shows the ticker, share count, current market value, and percent of portfolio; cash appears as the last row with a neutral color.

Below allocation is today's plan. Each decision from Run 1 appears as its own card showing an action label (buy, sell, or hold) colored by action type, the ticker or group of tickers it applies to, the share count if applicable, and the reasoning text. If the plan is empty or absent, a single placeholder card reads "no plan available for today."

Below the plan is Claude's briefing card. It contains a highlighted block showing the market summary at the top, followed by the daily lesson block which shows the term, an English explanation, and a Chinese explanation. The briefing card uses a subtle amber accent to visually separate the AI's narrative from the data-driven sections around it.

The trade history card comes next and is the most interactive section. Above the list sits a header row: on the left, the section label with a count of how many trades are currently shown; on the right, a button to export the current filtered set as a CSV file. Below the header sits a row of filter pills: this week, this month, this quarter, this year, all time, and by ticker. Selecting by ticker collapses the view to groups rather than a flat list, each group showing the ticker and the total realized profit across trades of that ticker. Inside the default flat list, each trade row shows the action in bold, the ticker, the share count, the price, the date on the right, and for sell trades an additional realized-profit pill indicating whether the round trip made or lost money.

The performance stats card is the penultimate section. It lays out a compact grid of labeled metric tiles: win rate, average winner versus average loser, maximum drawdown, daily volatility, best trade, and total realized profit. On phone, the grid becomes two columns instead of three.

The footer closes the page with the generated-at timestamp and the schema version, both in muted small text.

## 7. Visual Style

The color palette is light and clean. The background is a soft cool gray. Cards sit on pure white with subtle shadows. Text is very dark slate for primary content and medium slate for secondary. Positive values are a vivid emerald green; negative values are a clear red; neutral accents like action pills for hold or cash use warm amber or light gray. Interactive accents like active filter pills use deep navy for contrast.

Typography leans on the standard system font stack, with Inter preferred when available. Numeric values use tabular-nums so digits align in tables and comparisons. The hero portfolio value is a display-weight size, dropping to a regular heading scale inside card labels, body scale for values, and a small caption scale for muted timestamps and helper text.

Spacing is generous but not loose. Cards have fourteen pixels of internal padding on phone and eighteen on desktop. Cards are separated by twelve pixels of vertical spacing. Rows within cards use six pixels of padding with a subtle hairline separator between them.

The aesthetic is consistent across every section. There are no variant card styles, no exotic decorations, no dark-mode toggle in this phase.

## 8. Dashboard Components

The Vue application is organized around one root component and twelve child components, plus a small number of supporting composables and utilities. The root component is responsible for loading the JSON file, handling its loading and error states, and passing the resolved data down to the presentation components. It uses a composable that wraps the fetch call, exposes reactive data, loading, and error values, and offers a manual reload function for debugging.

Each visible section corresponds to its own component: the header, the hero, the equity chart, the allocation panel, the today-plan card, the briefing card, the trade-history card, the stats grid, and the footer. The trade-history card additionally delegates its filter pills to a small filter-tabs component, its flat row rendering to a trade-row component, and its by-ticker group rendering to a per-ticker-group component. Each component receives only the slice of data it needs and has no awareness of the global data shape beyond that.

Three supporting utility files exist. One holds a TypeScript interface describing the JSON contract so every component can type-check against it. One holds currency and percent formatting functions shared across the UI. One holds the client-side CSV generation used by the export button.

Styling is done entirely through Tailwind utility classes. A Tailwind configuration file defines the semantic color tokens (emerald as positive, red as negative, amber as neutral, deep slate as accent) so they can be changed in one place if the brand evolves.

## 9. Main Repository Changes

The main repository gains a single new file in its tools directory, the dashboard exporter. Its public surface is one function that reads the database path, the plan file path, and the output path and returns the fully-built dictionary after writing it. The function is pure in the sense that it has no side effects other than the atomic file write. It is also permissive: it never raises. Any internal failure results in a JSON file whose error field is set, so the dashboard can display a banner instead of looking frozen.

The Run 2 command in the main module gets one new step added at its end. After trade execution and the Telegram notification, it calls the exporter with the expected output path resolved from an environment variable (defaulting to the sibling dashboard repository's public folder), and then calls a small git helper that runs the stage, commit, and push sequence in that folder. Both operations are wrapped in exception handlers and log warnings on failure.

The old dashboard module, the old generated HTML output, and the tests that exercise the HTML generator are deleted outright. There is no transitional period, no dual-writing. The replacement is clean.

The existing engine, database, claude agent, notifier, risk rules, screener, and technical modules are untouched. The existing Run 1, monitor, history, and weekly digest commands are untouched.

No new Python dependencies are introduced in the main repository. The exporter uses only the standard library plus the existing stock data helpers, and the git helper wraps standard subprocess calls.

## 10. Testing Strategy

The main repository gets a new test file covering the exporter. Seven scenarios are covered: a happy path with a seeded database and a present plan file producing a valid JSON document with every expected top-level section; a missing plan file resulting in an empty today-plan section but all other sections populated; an empty database resulting in empty or zero-valued fields without any crashes; a set of hand-crafted closed round trips whose derived statistics match hand-calculated expectations to guard against math regressions; a mocked failing price fetch that results in a position with a null current price and no crash; an assertion that the write is atomic by checking the temp file mechanism; and a pinned schema version regression test that fails if the version is accidentally bumped without intent.

Existing Run 2 integration tests gain one new assertion: after running, the data JSON file is written to the expected path. A second assertion confirms that simulating an exporter failure does not prevent the trade execution steps from completing.

The dashboard repository uses Vitest. Eight scenarios are covered: the loading state renders the skeleton before fetch resolves; the error state renders the banner when the fetch rejects or the JSON is malformed; a newer schema version renders a warning banner but still shows content; an empty data document renders the empty state for each section without crashing; the filter tabs in the trade history produce the correct subsets for each time window; the CSV export produces rows matching a snapshot of expected output; the per-ticker grouping computes realized profit totals that match hand-calculated values; and the allocation percentages plus cash always sum to exactly one hundred after rounding.

Tests focus on logic and prop shapes rather than pixel-accurate visual regression. A full visual snapshot system is not introduced in this phase.

## 11. Error Handling Contract

Failures are handled at the edge closest to where they occur, and each failure mode has a defined behavior.

When the data file is missing in the dashboard repository, the dashboard shows an empty-state message telling the user to run the agent and reloads automatically every thirty seconds. When the data file is present but fails to parse, the dashboard shows an error banner and, if a previous good load was cached in localStorage, offers to display that stale view. When the schema version is higher than the dashboard was built against, a warning banner appears at the top of the page but the best-effort render continues below it.

When the main repository cannot find the dashboard repository folder, the exporter logs a warning and skips silently. When the exporter raises any internal exception, it writes a minimal JSON file whose only meaningful content is an error field describing what went wrong, and the dashboard displays that on load. When the git push fails after a successful export, the next run will retry; nothing blocks. When fetching a current price for a specific position fails, that position is included without a current price, its derived fields are left empty, and the allocation chart simply omits its slice rather than showing a broken percentage.

The controlling principle across all of this is that trade execution is sacred. Dashboard export failures, git failures, and price fetch failures never propagate up to halt or distort Run 2's trading behavior.

---

## 12. Deletions

This spec includes deletions. The existing dashboard generator module in the main repository is removed in full. The existing generated HTML output file is no longer produced and its directory entry is removed. The tests that validate HTML generation are removed and replaced by the new exporter tests. No backwards-compatibility shims, no migration period, no dual-writing — the replacement is clean.

---

## 13. Future Phases

The next phase, deferred to its own future spec, adds live hosting. The dashboard repository is connected to Cloudflare Pages for static hosting, protected by Cloudflare Access so only the user's email address can reach it. The main repository gains a step that uploads the data JSON to Cloudflare R2 after Run 2, and the dashboard repository is changed to fetch from that R2 URL instead of its local public folder. Because the data contract is unchanged, no other modifications are required.

A subsequent phase, further deferred, might introduce a proper remote database such as Cloudflare D1, Turso, or Supabase so multiple snapshots can be queried historically without relying on git history for archival. That phase would change how the JSON is produced but not its shape, so the dashboard would continue to work unmodified.

This design does not preclude either of those directions and was chosen precisely because it makes them cheap to reach.
